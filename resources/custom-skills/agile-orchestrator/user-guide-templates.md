# 用户引导卡片模板库

> 敏捷开发 Skill 体系的标准引导模板

---

## 模板 1：需求澄清助手

```markdown
## 🎯 需求澄清助手 已激活

**我能帮你**：
- 将模糊需求转成标准用户故事
- 生成验收标准（Given/When/Then 格式）
- 识别依赖、风险、待确认项
- 评估是否能进 Sprint

**使用方式**：
```
"用户能上传头像，支持裁剪和滤镜"
"我想做个积分系统，用户签到得积分"
"帮我分析这个需求：[粘贴需求描述]"
```

**执行流程**：
```
1. 你描述需求（1-2 句话或粘贴 PRD）
2. 我分析并输出用户故事 + AC + 风险项
3. 你确认或提出修改意见
4. 我生成最终版（可导入 Jira/飞书/禅道）
```

**预期耗时**：2-3 分钟

**输出内容**：
- 标准用户故事（As a... I want... So that...）
- 验收标准（3-8 条 AC）
- 风险项和待确认问题
- 可导入格式（JSON/Markdown）

**接下来你说**：直接描述你的需求即可
```

## 模板 2：任务拆解助手

```markdown
## 📋 任务拆解助手 已激活

**我能帮你**：
- 将需求/用户故事拆成可执行任务
- 估算故事点（S/M/L 或斐波那契数列）
- 识别任务依赖和阻塞关系
- 生成 Sprint Backlog 格式

**使用方式**：
```
"把这个需求拆解一下"
"这个任务多少故事点"
"生成下轮 sprint 的 backlog"
"帮我排个优先级"
```

**执行流程**：
```
1. 你提供需求或用户故事（或引用上文）
2. 我拆解成任务列表（粒度<1 天）
3. 我估算故事点 + 识别依赖
4. 你确认或调整
5. 我生成可导入格式
```

**预期耗时**：3-5 分钟

**输出内容**：
- 任务列表（含标题、描述、负责人）
- 故事点估算（含依据）
- 依赖关系图
- Sprint Backlog 格式（可导入）

**接下来你说**：提供要拆解的需求，或说"用上面的需求"
```

## 模板 3：代码审查助手

```markdown
## 🔍 代码审查助手 已激活

**我能帮你**：
- 审查代码质量和规范
- 识别安全风险和性能问题
- 检查测试覆盖和文档完整性
- 生成 PR 审查报告

**使用方式**：
```
"帮我 review 这段代码：[粘贴代码]"
"审查这个 PR：[粘贴 PR 链接或描述]"
"看看有没有安全问题"
"这个代码能合并吗"
```

**执行流程**：
```
1. 你提供代码或 PR 描述
2. 我分析代码（功能/质量/安全/性能）
3. 我生成审查报告 + 问题清单
4. 你根据建议修改或直接合并
```

**预期耗时**：2-5 分钟（取决于代码量）

**输出内容**：
- 审查结论（✅通过 / ⚠️需修改 / ❌拒绝）
- 问题清单（文件 + 行号 + 问题 + 建议 + 优先级）
- 优点和值得肯定的地方
- 最终建议

**接下来你说**：粘贴代码或 PR 描述
```

## 模板 4：测试验收助手

```markdown
## 🧪 测试验收助手 已激活

**我能帮你**：
- 根据需求生成测试用例
- 设计边界条件和异常场景
- 规划回归测试范围
- 生成验收报告模板

**使用方式**：
```
"生成测试用例"
"这个功能怎么验收"
"帮我设计边界测试"
"回归测试范围是什么"
```

**执行流程**：
```
1. 你提供需求或功能描述（或引用上文）
2. 我分析并生成测试点
3. 我设计测试用例（含预期结果）
4. 你确认或补充
5. 我生成测试文档
```

**预期耗时**：3-5 分钟

**输出内容**：
- 测试点清单（功能/边界/异常）
- 测试用例（含前置条件、步骤、预期结果）
- 回归测试范围建议
- 验收报告模板

**接下来你说**：提供要测试的功能描述
```

## 模板 5：开发实现助手

```markdown
## 💻 开发实现助手 已激活

**我能帮你**：
- 生成代码实现方案
- 编写具体代码示例
- 修改现有代码
- 生成 PR 描述和变更说明

**使用方式**：
```
"实现这个功能：[描述]"
"帮我写一个 xx 函数"
"修改这段代码：[粘贴代码]"
"生成 PR 描述"
```

**执行流程**：
```
1. 你描述功能需求或提供代码
2. 我分析并生成实现方案
3. 我编写代码示例
4. 你确认或提出修改
5. 我生成最终代码 + PR 描述
```

**预期耗时**：5-10 分钟

**输出内容**：
- 实现方案说明
- 代码示例（含注释）
- 变更说明
- PR 描述模板

**⚠️ 能力边界**：
- 我可以生成代码，但无法直接修改你的仓库
- 你需要手动应用代码变更
- 复杂修改建议先在测试环境验证

**接下来你说**：描述要实现的功能或粘贴代码
```

## 模板 6：发布复盘助手

```markdown
## 📊 发布复盘助手 已激活

**我能帮你**：
- 生成版本发布检查清单
- 编写发布公告草稿
- 组织迭代复盘会议
- 生成改进行动计划

**使用方式**：
```
"生成发布检查清单"
"写个版本公告"
"这轮 sprint 复盘一下"
"生成改进计划"
```

**执行流程**：
```
1. 你提供版本信息或迭代情况
2. 我分析并生成检查清单/复盘框架
3. 你补充实际情况和问题
4. 我生成完整报告 + 改进行动
```

**预期耗时**：5-8 分钟

**输出内容**：
- 发布检查清单 / 复盘报告
- 做得好的地方（Keep）
- 需要改进的地方（Problem）
- 行动计划（Try）

**接下来你说**：描述版本或迭代情况
```

## 通用异常处理模板

### 需求不清晰

```markdown
## ⚠️ 需要更多信息

**问题**：需求描述不够清晰，我无法准确分析

**请补充**（选一个）：
1. 这个功能的**目标用户**是谁？
2. 用户用这个功能要完成**什么任务**？
3. 有没有**参考案例**或竞品？

**或者你可以说**：
- "先按你的理解写一版"
- "跳过这个需求"
- "我换个需求"
```

### 超出能力范围

```markdown
## ⚠️ 超出我的能力范围

**我能做的**：
- 需求分析、任务拆解、代码审查、测试设计

**我做不了的**：
- 直接修改代码仓库（需要 Git 集成）
- 执行测试用例（需要测试环境）
- 部署上线（需要 CI/CD 权限）

**建议**：
- 我可以生成修改方案和代码示例，你手动应用
- 我可以生成测试用例，你在测试环境执行

**你还想继续吗？**
```

### 上下文丢失

```markdown
## ⚠️ 未找到上下文

**问题**：我没有找到相关的需求或任务上下文

**你可以**：
1. 重新描述需求
2. 引用上文的消息
3. 说"重新开始"开启新流程

**你想怎么做？**
```
